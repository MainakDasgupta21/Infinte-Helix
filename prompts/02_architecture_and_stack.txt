is named "Infinite_helix" and contains two main directories: backend (Python
Flask REST API) and frontend (React 18 SPA).

--------------------------------------------------------------------------------
DATA FLOW ARCHITECTURE
--------------------------------------------------------------------------------

The system follows a standard SPA-to-REST architecture:

1. FRONTEND (React SPA on port 3000)
   └─→ Axios HTTP client with Firebase ID token in Authorization header
       └─→ BACKEND (Flask REST API on port 5000)
           ├─→ Firebase Firestore (primary persistence)
           │   └─→ SQLite LocalStore fallback (when Firestore unavailable)
           ├─→ HuggingFace Transformers (emotion/sentiment inference)
           │   └─→ Demo mock classifiers fallback
           ├─→ Groq API (chatbot LLM completions)
           │   └─→ Template-based responses fallback
           ├─→ Google Calendar API (OAuth + meeting fetch)
           ├─→ Microsoft Graph API (OAuth + meeting fetch)
           ├─→ pynput (keyboard/mouse monitoring in background thread)
           ├─→ psutil (screen/process tracking)
           └─→ plyer (desktop notification toasts)

2. STANDALONE STRESS AGENT (optional)
   stress_agent.py (pynput listener)
   └─→ POST /api/stress/ingest every 10 seconds
       └─→ Frontend useStressDetector hook polls /stress/latest

3. CLIENT-SIDE ONLY (localStorage)
   - Period/cycle entries: helix_period_entries
   - Daily cycle logs: helix_cycle_daily_logs
   - Period duration: helix_period_duration
   - Pregnancy data: helix_pregnancy_data
   - Task mode: helix_task_mode
   - Pregnancy setup flag: helix_pregnancy_setup_done
   - Theme preference: helix_theme
   - Meal reminder settings: helix_meal_reminders

4. POLLING INTERVALS
   - WellnessContext: polls /dashboard/today and /tracker/status periodically
   - useStressDetector: polls /stress/latest every few seconds
   - useBreakReminder: timer-based interval checks

--------------------------------------------------------------------------------
COMMUNICATION PROTOCOL
--------------------------------------------------------------------------------

- All API endpoints are under /api/* prefix
- CORS allows all origins ("*") with credentials support
- Authentication: Firebase ID token as Bearer token in Authorization header
- Content-Type: application/json for all requests
- Backend returns JSON responses with consistent error shapes
- Demo mode: accepts any Bearer token, generates local user IDs
